> Working name: **LastBite**. A surplus-food rescue marketplace: local hotels/restaurants/bakeries list food that would otherwise be thrown out at end of day; nearby customers claim it cheap and pick it up. Prices auto-drop as closing time nears, and a claim locks the item for a short, provider-set pickup window.

---

## What we're building

Restaurants throw away unsold perishable food daily. LastBite turns that waste into revenue: a provider taps "I have 8 sandwiches and 5 boxes of mithai left," nearby customers see it, claim a bag at a discount, and walk over to collect it.

Two differentiators vs. plain CRUD: prices **decay automatically** as closing time approaches, and claims create a **time-boxed lock** that auto-releases if the customer doesn't show. Both are implemented as Inngest durable workflows. Closest real product: Too Good To Go (not in India — that's the opening).

## Two sides

- **Provider (the "cook"):** hotel/restaurant/bakery. Registers a business, publishes listings, gets notified the instant a bag is claimed, verifies the pickup code on collection. Multi-tenant — each business is its own Better Auth org.
- **Customer:** browses nearby deals **as a guest** (no login required to browse), verifies via **phone OTP only at the moment of claiming**, picks up within the window. Can "follow" a provider for new-deal alerts.

No real payments — **"reserve now, pay at pickup."**

## Tech stack (pinned)

- **Next.js 16** — App Router, React 19, Turbopack, **React Compiler enabled** (`reactCompiler: true` in `next.config.ts`). Use Server Components + Server Actions; client components only where interactivity is required (map, countdown, claim button).
- **TypeScript** strict mode.
- **Neon Postgres** + **Drizzle ORM** + **Drizzle Kit** for migrations. Money stored as Postgres `numeric` — never JS floats.
- **Better Auth** (self-hosted, Drizzle adapter):
  - `organization` plugin → each provider is an org; staff are members with roles.
  - `phoneNumber` plugin → OTP login for customers.
  - Auth tables (`user`, `session`, `account`, `organization`, `member`) are **generated by the Better Auth CLI — do not hand-write or edit them**.
- **Inngest** → all durable workflows (see `docs/inngest-workflows.md`).
- **Resend** for email. Skip SMS/WhatsApp for v1. In dev, log notifications to console/DB.
- **Tailwind + shadcn/ui** for customer-facing UI.
- Deploy on **Vercel**.

## Architecture

One Next.js app, two route groups:
- `app/(customer)/` — browse, listing detail, claim flow, my-pickups
- `app/(provider)/` — dashboard, new listing, incoming claims, verify pickup
- `app/api/inngest/route.ts` — Inngest serve handler

Auth gates routes by role: phone-verified customers vs. provider org members.

All mutations go through **Server Actions** → write to Postgres → `inngest.send(...)` event → Inngest workflow handles async/time-based work.

### Data fetching split

| Data | Pattern |
|---|---|
| Static reads | `async` Server Components querying Drizzle directly — no Route Handler, no React Query |
| Live reads (decaying price, nearby feed) | TanStack Query against small Route Handlers (`GET /api/listings/nearby`, `GET /api/listings/:id/status`) with `refetchInterval` ~20–30s |
| Writes | Server Actions with `useActionState` (React 19); claim button uses `useOptimistic` for instant UI flip |

Do **not** use React Query for non-live data. Do **not** use Server Actions for reads.

The geo feed is a client component (browser Geolocation API coords go in the React Query key). Server-render an initial unlocated list to avoid blank screen.

### Three Inngest workflows (full spec in `docs/inngest-workflows.md`)

1. **`reservationHold`** ← triggered by `reservation/created`
   Atomically locks stock (`UPDATE ... WHERE quantity_available >= quantity` — the only place stock is decremented), notifies cook + customer, then `step.waitForEvent` for pickup confirmation with `timeout = clampedWindow`. On timeout: releases stock, marks reservation `expired`, re-offers the listing. **This is the centerpiece — demo it.**

2. **`priceDecay`** ← triggered by `listing/created`
   Walks `decaySchedule` tiers with `step.sleepUntil`, drops `current_price` at each tier, then marks listing `expired` at `closeAt` if stock remains.

3. **`notifyNewDeal`** ← triggered by `listing/created`
   Fans out alerts to followers of the provider.

### Key constraint: pickup window

Provider sets default/max window in minutes. App **always clamps to `GLOBAL_MAX_PICKUP_MINUTES = 30`**. Never trust a client-supplied value.

## Data model (`src/db/schema.ts`)

App tables: `providers`, `listings`, `reservations`, `follows`, `notifications`.

Key design decisions:
- `providers.orgId` — FK to Better Auth organization; enables multi-tenant staff management.
- `listings.decaySchedule` — `jsonb` array of `{ afterMinutes, discountPct }` tiers.
- `listings.closeAt` — hard cutoff timestamp; `priceDecay` workflow sleeps until it.
- `reservations.pickupCode` — 4-digit code shown to customer, verified by provider.
- `reservations.holdExpiresAt` — set by `reservationHold` after clamping the window.

Stock integrity: **only** the conditional `UPDATE` in `reservationHold` decrements `quantity_available`. No other code path touches it.

## Conventions

- Server Actions for all writes; Zod validation at every Server Action boundary.
- All Inngest event names: `domain/thing.happened` format.
- Every `step.run` id must be **stable and unique** within its function (Inngest uses these for memoization on retries).
- Provider dashboard: plain/functional. Customer claim flow: polish here — it's what gets demoed.

## Build phases (ship in order)

1. **Skeleton + auth** — Drizzle + Neon connected, Better Auth org + phoneNumber plugins, provider registers, customer verifies phone.
2. **Listings** — Provider creates listing → DB → `inngest.send("listing/created")`. Customer sees nearby list.
3. **The lock** — Wire `reservationHold`. Test both paths: successful pickup AND timeout auto-release (watch in Inngest dashboard).
4. **Price decay** — Wire `priceDecay`; use short tiers (1m/2m) in dev to watch drops live.
5. **Follows + alerts** — `notifyNewDeal` fan-out + "follow a provider" button.
6. **Polish** — PWA, geo radius filter, "meals rescued / money saved" stat, README screenshots.
